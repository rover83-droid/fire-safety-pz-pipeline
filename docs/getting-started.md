# Инструкция для новичка: как собрать раздел пожарной безопасности

Эта инструкция написана для инженера, который **не программист**. Она проведёт
вас от установки до готового DOCX-файла раздела 9 «Мероприятия по обеспечению
пожарной безопасности» (ПЗ МПБ).

> Главное, что нужно понять сразу: система **не выдумывает нормы**. Каждая цитата
> из СП/СНиП проверяется дословно по тексту, который вы сами загрузили. Если
> данных не хватает — она честно скажет об этом, а не подставит выдуманное
> значение. Поэтому итоговому документу можно доверять.

---

## 1. Что вам понадобится (один раз)

1. **Claude Code** (или Codex) — среда, в которой вы общаетесь с системой обычными
   фразами. Установка: [docs.claude.com/claude-code](https://docs.claude.com/claude-code).
2. **Python 3.11 или новее** — нужен, чтобы работали встроенные проверки. Проверить,
   установлен ли он, можно командой `python --version`.
3. **Тексты нормативных документов** (СП, СНиП, ФЗ-123) в виде файлов `.md` или
   `.txt`. Система работает только с теми документами, которые вы дадите.

Установка самого продукта (один раз, в терминале):

```powershell
git clone https://github.com/rover83-droid/fire-safety-pz-pipeline.git mpb-pz
cd mpb-pz
pip install -e .[docx]
```

## 2. Два способа работать

- **Разговор (рекомендуется).** Вы пишете системе фразами: «Начать новый проект»,
  «Запусти агента 1», «Собери финальный DOCX». Система сама выполняет команды и
  показывает короткие карточки статуса. Так работает большинство инженеров.
- **Команды (CLI).** Те же действия можно вызвать вручную. Ниже показаны обе формы,
  чтобы вы понимали, что происходит «под капотом». Все команды в PowerShell
  предваряются один раз за сессию: `$env:PYTHONPATH='src'`.

Дальше — сквозной пример для вымышленной **блочно-модульной котельной (Ф5.1)**.

## 3. Шаг за шагом

### Шаг 1. Новый проект и паспорт объекта

Скажите системе: **«Начать новый проект ПЗ МПБ»** и отвечайте на её вопросы —
класс функциональной пожарной опасности, степень огнестойкости, категория,
этажность, высота, объём, площади и т.д. Это «паспорт объекта».

Если каких-то данных нет — так и скажите. Система пометит их как **лакуну**
(пробел) и не даст двигаться дальше, пока критичные поля не заполнены. Это
защита: без категории помещений нельзя честно подобрать нормы.

CLI-эквивалент проверки готовности:
```powershell
python -m mpb_pz_flow.cli status --project-dir .\projects\bmk
```
Стадия `passport_ready` означает: паспорт готов, можно подбирать нормы.

### Шаг 2. Загрузка нормативных документов (корпус)

Каждый СП нужно один раз «зарегистрировать» — система нормализует кодировку,
посчитает контрольную сумму и запомнит редакцию:

```powershell
python -m mpb_pz_flow.cli corpus-add --project-dir .\projects\bmk `
  --file "C:\путь\к\СП 8.13130.md" --document "СП 8.13130" --edition-year 2020 --status добровольный
```

Проверить, что всё на месте и не повреждено:
```powershell
python -m mpb_pz_flow.cli corpus-verify --project-dir .\projects\bmk
```

### Шаг 3. Инженерные расчёты

Система умеет считать простые нормативные величины (расход воды на пожаротушение,
ширину проездов и т.п.) — по формулам из таблиц СП, с указанием основания:

```powershell
python -m mpb_pz_flow.cli calc-run --project-dir .\projects\bmk
python -m mpb_pz_flow.cli calc-list   # список доступных калькуляторов
```
Если данных для расчёта не хватает или объект выходит за таблицу — калькулятор
**откажется считать**, а не подставит примерное число.

### Шаг 4. Подбор норм по разделу (Агент 1)

Скажите: **«Запусти агента 1»**. Система читает загруженные СП и подбирает нормы,
относящиеся к вашему объекту, с **дословными** цитатами и точными пунктами.

Если вы работаете через файл (или это делает агент), нормы устанавливаются
командой **`norms-add`** (см. раздел 4 ниже) — она сразу проверит цитаты.

### Шаг 5. Матрица применимости и черновик (Агент 2)

Скажите: **«Запусти агента 2»**. Система:
- строит матрицу — по каждой норме показывает, **применима ли она** к вашему
  объекту, и на основании какого поля паспорта (`build-matrix`);
- собирает черновик текста раздела (`assemble`).

### Шаг 6. Аудит (Агент 3)

Скажите: **«Запусти аудит»**. Проверяются: дословность цитат, обоснованность всех
чисел, полнота раздела, стиль. Если что-то не так — система вернёт список ошибок с
указанием, что и где поправить, и **не даст закрыть раздел**.

```powershell
python -m mpb_pz_flow.cli audit --project-dir .\projects\bmk --section "Категорирование"
python -m mpb_pz_flow.cli gate  --project-dir .\projects\bmk --section "Категорирование"
```
`gate` — это «действительно ли готово». Он ещё и проверяет, что после аудита файлы
никто не менял (защита от подмены).

### Шаг 7. Готовый документ

Скажите: **«Собери финальный DOCX»**. Получите файл, оформленный по ГОСТ
(Times New Roman, поля, титульный лист, нумерация, перечни сокращений и НД):

```powershell
python -m mpb_pz_flow.cli export-docx --project-dir .\projects\bmk --section "Категорирование" --front-matter
```

## 4. Две новые команды

### `norms-add` — установить нормы из файла с проверкой

Когда нормы раздела подготовлены в виде файла (по одной норме в строке, формат
JSONL), команда их проверяет и устанавливает:

```powershell
python -m mpb_pz_flow.cli norms-add --project-dir .\projects\bmk `
  --section "Категорирование" --file .\normy.jsonl
```

Что она делает:
- проверяет, что у каждой нормы заполнены обязательные поля (иначе откажет);
- **сразу сообщает, какие цитаты не найдены в источнике** — вы чините их до
  аудита, а не узнаёте об ошибке в конце;
- добавляет новые нормы, а совпадающие по `norm_id` — обновляет на месте (без
  дубликатов). Флаг `--replace` полностью заменяет набор норм раздела.

Одна строка файла `normy.jsonl` выглядит так (пример):
```json
{"norm_id": "sp12-5-2", "document": "СП 12.13130", "edition_year": 2009, "point": "п. 5.2", "quote": "Определение категорий помещений следует осуществлять путем последовательной проверки...", "subject": "порядок определения категорий", "trigger_parameter": "production_room_categories", "source_file": "standards/sp-12-13130-2009.md"}
```

### `consolidate` — собрать весь том 9 в один документ

Когда отдельные разделы готовы и прошли аудит, эта команда собирает из них единый
том 9 в порядке ПП РФ №87:

```powershell
python -m mpb_pz_flow.cli consolidate --project-dir .\projects\bmk
```

Что она делает:
- берёт содержательные разделы из проверенных `final.md` (их подделать нельзя —
  они хэш-привязаны);
- «Общие положения» и «Характеристику объекта» строит из паспорта;
- добавляет перечень сокращений и перечень нормативных документов;
- для ещё не разработанных разделов вставляет **честный плейсхолдер**
  «заполняется проектной организацией»;
- сохраняет `tom9_svod.md` и `tom9_svod.docx` (ГОСТ).

Флаг `--no-docx` — если нужен только текст (Markdown), без DOCX.

## 5. Полная сборка одной кнопкой (мультиагентный сценарий)

Для опытных пользователей есть сценарий оркестрации `workflows/tom9.workflow.js`:
он запускает по одному агенту на каждый раздел тома **параллельно**, затем
**механически** (кодом, не «на глаз») проверяет каждый раздел и собирает сводный
том. Запускается только по явной команде — он расходует много ресурсов.

Логика сценария (4 фазы):
1. заморозка паспорта (одна версия исходных данных);
2. параллельный подбор норм по разделам;
3. механический гейт по каждому разделу;
4. маршрутизация правок (до 3 попыток) и сборка тома.

## 6. Что делать, если система «ругается»

- **`quote_unverified`** — цитата не совпала с источником дословно. Откройте СП,
  скопируйте точную формулировку (можно с пропуском `...` в середине).
- **`number_unverified`** — в тексте есть число без обоснования. Оно должно быть в
  паспорте, расчёте или цитате нормы.
- **«Артефакт изменён после аудита»** — файл поменяли после проверки. Просто
  запустите аудит заново.
- **Лакуна / «источник отсутствует»** — не хватает исходных данных. Это не ошибка
  системы, а честный сигнал: нужны данные от вас или смежного раздела.

Никакую из этих проверок нельзя «уговорить» пропустить недостоверные данные —
в этом и смысл продукта.
