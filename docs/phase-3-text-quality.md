# Фаза 3 — качество производственного текста

Цель: текст ПЗ соответствует производственному инженерному стилю (эталон —
ВНИИПО) не по ощущению аудитора, а по механическим проверкам. Модуль: `style.py`.

## 1. Грамматики типов абзацев (`validate_paragraph_types`)

Маркер `[Тип X]` обязывает абзац соответствовать формуле типа (ошибка
`draft.type_grammar`):

| Тип | Формула | Проверка |
|---|---|---|
| А | Решение по норме | «в соответствии с …» + глагол решения (принято/предусмотрено/обеспечивается…) |
| Б | Отрицательное решение | отрицательная конструкция (не подлежит/не требуется/не нормируется…) |
| В | Системное решение | предмет (система/установка/алгоритм/оборудование/сигнал) |
| Г | Расчёт | число с единицей измерения + расчётная лексика (расчет/принят/составляет/=) |

## 2. Стилевой линтер (`validate_style`)

- `draft.style_future` (error) — «будет/планируется/предполагается»: намерение
  вместо решения;
- `draft.style_vague_citation` — «согласно требованиям норм» и т.п.: error без
  конкретного документа в абзаце, warning при его наличии;
- `draft.style_sp_without_number` (error) — «СП» без номера документа;
- `draft.style_missing_edition_year` (warning) — «СП 8.13130» без года редакции.

## 3. Полнота секций (`validate_section_content`)

`SECTION_REQUIREMENTS` расширен на все 17 разделов ПП РФ № 87 (Раздел 9 ПД);
варианты формулировок темы — через `|` («МГН|маломобильн»). Пропуск обязательной
темы — warning `draft.section_content_missing` (формулировки тем могут законно
отличаться, но пропуск виден в протоколе аудита и валит `validate --strict`).
Для полнотомных черновиков Ф5.1 дополнительно проверяется `required_content`
каждого раздела эталонной структуры.

## 4. Жизненный цикл лакун (`validate_lacunae`)

- `decisions.lacunae[]` — структурные записи `{parameter, status, note}`;
  запись со статусом, отличным от resolved/closed/снята, упоминающая документ,
  уже зарегистрированный в корпусе, — warning `decisions.lacuna_stale`;
- предложение черновика, заявляющее «источник … отсутствует в проектной базе»
  для документа, который есть в корпусе, — warning `draft.lacuna_stale`.

Это замыкает цикл: источник появился → конвейер сам подсказывает, какие
лакуны и абзацы устарели.

## 5. Включение в конвейер

- стадия `draft_ready` требует отсутствия ошибок грамматики типов и стиля;
- Уровень 2 аудита включает все проверки Фазы 3;
- warnings не блокируют гейт, но валят `validate --strict`.

## Результат на реальных данных (проект hvo)

- `draft.lacuna_stale`: черновик КиОПР заявляет «локальный нормативный источник
  по СП 44.13330 отсутствует в проектной базе», при том что СП 44.13330.2011
  зарегистрирован в корпусе — ровно тот случай, что был предсказан в ревью;
- грамматика типов: 14 находок в секции fire_brigade — абзацы, помеченные
  [Тип Б], на деле являются положительными выводами по порогу (нет отрицательной
  конструкции), а [Тип А] без «в соответствии с …»;
- стиль: «СП 8.13130» и «СП 4.13130» упомянуты без года редакции.
