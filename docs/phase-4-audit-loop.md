# Фаза 4 — петля аудита: маршрутизация и лимит итераций

Цель: замечания аудита возвращаются адресно нужному агенту, цикл правок
ограничен, незакрываемые дефекты эскалируются пользователю, а не крутятся
бесконечно. Модули: `routing.py`, петля в `pipeline.audit_project`.

## 1. Адресная маршрутизация (`routing.py`)

Каждая находка в `audit_report.json` несёт поле `route`:

| route | Возврат | Коды |
|---|---|---|
| `agent_1` | Агент 1 (разведка) | `passport.*`, `norms.*`, `decisions.*`, `corpus.*` |
| `agent_2` | Агент 2 (сборка) | `matrix.*`, `draft.*`, `volume_structure.*` |
| `orchestrator` | Решает оркестратор | `draft.number_unverified` (дефект паспорта или переноса), `agent_findings.invalid` |

Сводка — `routing_summary` в отчёте; CLI `audit` печатает её и группирует
находки по маршруту.

## 2. Лимит итераций и эскалация

Петля хранится в `audit_loop.json` набора секции (пишет только конвейер):

- провальный аудит **после изменения артефактов** увеличивает счётчик
  итерации; повторный прогон без правок — та же попытка (сравнение по
  SHA-256 из привязки отчёта);
- максимум **3 итерации**: на третьей провальной вердикт становится
  `ЭСКАЛАЦИЯ ПОЛЬЗОВАТЕЛЮ`, в отчёт добавляются `persistent_findings`
  (коды, не закрытые ни одной итерацией) и `iteration_history`
  (сводный протокол попыток);
- успешный аудит сбрасывает петлю.

Оркестратор обязан показать сводный протокол пользователю и остановиться;
автоматическое закрытие конвейера при эскалации запрещено (гейт и так не
пройдёт: `passed: false`).

## 3. Процедура цикла (для оркестратора)

1. `cli audit [--section …]` → при провале раздать находки: `agent_1` —
   Агенту 1, `agent_2` — Агенту 2 (каждому только его часть протокола).
2. После правок — полный повторный аудит (Уровень 1 → Уровень 2).
3. После 3-й неудачи — эскалация пользователю со сводным протоколом.

## Покрытие тестами

- маршрутизация по префиксам и исключениям;
- итерация не тратится на прогон без правок;
- три провала с правками → эскалация, `persistent_findings` содержит
  незакрытый код;
- успешный аудит сбрасывает петлю и открывает стадию `audit_passed`.
