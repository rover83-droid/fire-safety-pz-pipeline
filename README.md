# MPB PZ Flow — конвейер для раздела «Мероприятия по обеспечению пожарной безопасности»

> **English summary:** an auditable, artifact-first pipeline that helps fire-safety engineers draft and verify the Russian project-documentation chapter "Мероприятия по обеспечению пожарной безопасности" (fire-safety explanatory note) through a Claude Code / Codex dialog. Every cited norm is mechanically checked against a local source text; nothing is invented. See [docs/architecture.md](docs/architecture.md) for the technical design.

## Что это и для кого

Это инструмент для инженеров пожарной безопасности, которые готовят раздел 9 проектной документации («Мероприятия по обеспечению пожарной безопасности», ПЗ МПБ) по требованиям ПП РФ №87.

Вы общаетесь с системой обычными фразами на русском языке — через диалог в **Claude Code** (или Codex). Никакого программирования не требуется. Система сама:

- ведёт «паспорт» объекта (класс функциональной пожарной опасности, степень огнестойкости, площади, объёмы и т.д.) и фиксирует, какие данные подтверждены, а какие отсутствуют;
- извлекает нормативные требования **только из реальных текстов СП/СНиП/ФЗ**, которые вы загрузили в проект, и проверяет каждую цитату дословно — придумывать нормы система не умеет и не позволяет;
- считает применимость требований и инженерные величины (расходы воды, ширины проездов и т.п.) по формулам из нормативных таблиц, а не «на глаз»;
- собирает черновик текста по канонической структуре раздела и проверяет его на полноту, стиль и корректность ссылок;
- проводит аудит и не даёт собрать итоговый документ, пока не устранены ошибки;
- выгружает готовый раздел в DOCX, оформленный по ГОСТ (Times New Roman, поля, подписи таблиц, титульный лист, перечни сокращений и нормативных документов).

**Главное правило системы:** если данных не хватает — она честно сообщает об этом как о пробеле («лакуне»), а не выдумывает значения. Это сделано специально, чтобы документу можно было доверять.

## Зачем это нужно

Подготовка ПЗ МПБ — рутинная, но критически важная по точности работа: ошибка в ссылке на пункт СП или неверная категория помещения может стоить отказа в экспертизе. Этот инструмент берёт на себя механическую часть проверки (цитаты, применимость, числа, структуру), оставляя инженеру инженерные решения.

## Быстрый старт (без знания кода)

1. Установите [Claude Code](https://docs.claude.com/claude-code) или Codex CLI и подключите эту папку как проект.
2. Установите Python 3.11+ (нужен один раз, чтобы запускать встроенные проверки).
3. Откройте диалог и напишите:

   ```text
   Начать новый проект ПЗ МПБ
   ```

4. Дальше просто отвечайте на вопросы системы. Типичный разговор выглядит так:

   ```text
   Начать новый проект ПЗ МПБ
   Покажи статус
   Что не хватает для запуска агента 1?
   Запусти агента 1
   Запусти агента 2
   Запусти аудит
   Собери финальный DOCX
   ```

5. Когда аудит пройден, попросите: `Собери финальный DOCX` — получите готовый файл с оформлением по ГОСТ.

Подробное пошаговое руководство для новичка (установка → готовый DOCX, включая
команды `norms-add` и `consolidate`): [docs/getting-started.md](docs/getting-started.md).
Разговорный сценарий работы: [docs/codex-native-workflow.md](docs/codex-native-workflow.md).

## Как устроена надёжность

Систему специально проектировали так, чтобы ей нельзя было «подсунуть» недостоверный документ:

| Гарантия | Как обеспечена |
|---|---|
| Нормы не выдуманы | Каждая цитата дословно сверяется с зарегистрированным текстом источника (`cli corpus-add` + `cli corpus-verify`) |
| Применимость не на глаз | Машиночитаемые условия (`triggers`) вычисляют статус из паспорта объекта; ручное переопределение требует обоснования |
| Числа в тексте обоснованы | Каждое число с единицей измерения должно быть прослеживаемо до паспорта, расчёта или цитаты нормы |
| Итоговый отчёт не подделать | `audit_report.json` создаётся только конвейером и привязан хэшами к проверенным файлам — любое изменение после аудита аннулирует вердикт |
| Пробелы в данных видны, а не скрыты | Недостающие инженерные данные фиксируются как «лакуны» со статусом, а не заполняются предположениями |

Подробности по каждому пункту — в `docs/phase-0-integrity.md` … `docs/phase-5-export.md`.

### Структура проекта

```text
projects/<name>/
  state.json
  artifacts/
    passport.json                  # паспорт объекта (один на проект)
    decisions.json                 # журнал решений и принятых редакций норм
    audit_report.json              # вердикт аудита, привязанный хэшами
    sections/
      index.json                   # реестр секций
      <slug>/                      # отдельный набор артефактов на каждую секцию
        norms.jsonl
        applicability_matrix.jsonl
        draft.md
        final.md / final.docx
```

### Команды CLI (для тестов и восстановления; основной интерфейс — диалог)

```powershell
python -m mpb_pz_flow.cli init --project-dir .\projects\shop --fkp F3.1 --section "Наружное ВПС и проезды" --object-name "Магазин"

python -m mpb_pz_flow.cli gate --project-dir .\projects\shop
python -m mpb_pz_flow.cli export-docx --project-dir .\projects\shop --front-matter
```

### Мультиагентная сборка (fan-out по разделам + механический гейт)

Две команды поддерживают оркестрацию: агент собирает по разделу на субагента,
затем код (не агент) проверяет каждый раздел, а `consolidate` сводит том 9.

```powershell
# валидировать и установить нормы раздела из JSONL (проверяет обязательные поля
# и сразу сообщает, какие цитаты не верифицированы — до аудита)
python -m mpb_pz_flow.cli norms-add --project-dir .\projects\shop --section "Категорирование" --file .\norms.jsonl

# собрать сводный том 9 (порядок ПП-87) из проверенных final.md разделов;
# «Общие положения» и «Характеристика объекта» — из паспорта, обвязка — из редакций
python -m mpb_pz_flow.cli consolidate --project-dir .\projects\shop
```

Пример оркестрирующего сценария — `workflows/tom9.workflow.js` (фазы:
заморозка паспорта → fan-out разделов → механический гейт → сводный том).

## Браузерный GUI (устарел)

Поддерживаемые интерфейсы — диалог Claude Code/Codex и CLI. Браузерный GUI (`python .\run_gui.py`) по-прежнему работает с теми же артефактами, но заморожен: новые функции в него не добавляются.

## Важное правило безопасности

Нормативный текст должен поступать только из локальных файлов проекта или из проверенной вручную записи источника. Экстрактор и валидаторы не допускают тихого попадания выдуманных норм в текст — все цитаты механически проверяются против зарегистрированного корпуса.
